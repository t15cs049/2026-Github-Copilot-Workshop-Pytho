# ポモドーロタイマーカスタマイズ - Issue起票ガイド

## 概要
このドキュメントは、ポモドーロタイマーのカスタマイズに関する3つのissueを起票するためのガイドです。

## 作成されたIssueテンプレート

以下の3つのissueテンプレートが `.github/ISSUE_TEMPLATE/` ディレクトリに作成されています：

1. **pattern-a-visual-feedback.md** - 視覚的フィードバックの強化
2. **pattern-b-customization.md** - カスタマイズ性の向上
3. **pattern-c-gamification.md** - ゲーミフィケーション要素の追加

## Issue起票手順

### 方法1: GitHubのWeb UIを使用

1. GitHubリポジトリのページにアクセス
2. 「Issues」タブをクリック
3. 「New Issue」ボタンをクリック
4. 表示されるテンプレート選択画面から以下のいずれかを選択：
   - 「パターンA：視覚的フィードバックの強化」
   - 「パターンB：カスタマイズ性の向上」
   - 「パターンC：ゲーミフィケーション要素の追加」
5. テンプレートの内容を確認・必要に応じて編集
6. 「Submit new issue」をクリック

### 方法2: GitHub CLIを使用

```bash
# パターンA: 視覚的フィードバックの強化
gh issue create --template pattern-a-visual-feedback.md

# パターンB: カスタマイズ性の向上
gh issue create --template pattern-b-customization.md

# パターンC: ゲーミフィケーション要素の追加
gh issue create --template pattern-c-gamification.md
```

### 方法3: 直接URLからアクセス

以下のURLをブラウザで開くと、各テンプレートを使用したissue作成画面が開きます：

- パターンA: `https://github.com/[owner]/[repo]/issues/new?template=pattern-a-visual-feedback.md`
- パターンB: `https://github.com/[owner]/[repo]/issues/new?template=pattern-b-customization.md`
- パターンC: `https://github.com/[owner]/[repo]/issues/new?template=pattern-c-gamification.md`

※ `[owner]` と `[repo]` は実際のリポジトリ情報に置き換えてください。

## 各パターンの概要

### パターンA: 視覚的フィードバックの強化
**目的**: 視覚的な没入感がユーザーの集中力に与える影響を測定

**主な機能**:
- 円形プログレスバーのアニメーション
- 時間経過に応じた色の変化（青→黄→赤）
- 背景パーティクルエフェクトや波紋アニメーション

**技術スタック**:
- CSS3アニメーション
- Canvas API
- requestAnimationFrame

### パターンB: カスタマイズ性の向上
**目的**: 個人の好みに合わせた設定がユーザー継続率に与える影響を測定

**主な機能**:
- 時間設定の柔軟化（15/25/35/45分）
- テーマ切り替え（ダーク/ライト/フォーカスモード）
- サウンド設定（開始音/終了音/tick音）
- 休憩時間カスタム（5/10/15分）

**技術スタック**:
- LocalStorage/IndexedDB
- Web Audio API
- CSS Custom Properties

### パターンC: ゲーミフィケーション要素の追加
**目的**: ゲーミフィケーション要素がモチベーション維持と継続利用に与える影響を測定

**主な機能**:
- 経験値システムとレベルアップ
- 達成バッジ（3日連続、今週10回完了など）
- 週間/月間統計とグラフ表示
- ストリーク表示（連続日数カウント）

**技術スタック**:
- IndexedDB
- Chart.js/D3.js
- データ集計・可視化

## Issue間の関連性

3つのパターンは独立して実装可能ですが、以下の点で関連があります：

1. **データ管理**: パターンBとCは両方ともデータの永続化が必要
2. **UI/UX**: すべてのパターンが統一されたデザインシステムを共有すべき
3. **パフォーマンス**: パターンAとCは特にパフォーマンスに注意が必要

## 推奨される実装順序

1. **パターンB（カスタマイズ性）**: 基盤となる設定システムを先に構築
2. **パターンA（視覚的フィードバック）**: 視覚要素の追加
3. **パターンC（ゲーミフィケーション）**: データ蓄積が必要なため最後

ただし、各パターンは独立して実装可能なため、並行開発も可能です。

## 注意事項

- 各issueには適切なラベルが自動で付与されます
- 実装前にデザインモックアップの作成を推奨
- A/Bテストの計画を立てて効果測定を実施
- アクセシビリティとパフォーマンスに常に注意
